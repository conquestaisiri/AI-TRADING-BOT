Major problem 2: settings are loaded too aggressively

In config/settings.py, it loads required env variables immediately at import time.

That means the project will crash very early if .env is missing.

That is not always bad, but the way it is done is slightly rigid.

Why this is an issue

Sometimes you want:

a clean startup check

easier unit testing
the ability to inspect config state before everything explodes

Right now the code is basically saying:

That works, but it is a bit harsh and brittle.

Better approach

A settings object that validates on startup rather than during class attribute evaluation would be cleaner.

Not a fatal flaw.

But not polished.

Major problem 3: the strategy logic is too naive

This is the biggest real trading weakness.

The strategy is basically:

1h EMA20 vs EMA50 = trend
15m close above recent swing high + above average volume = long
15m close below recent swing low + above average volume = short

That is very simple.

Too simple, honestly.

Why this matters

A bot like this can fire on:

bad breakouts
noisy candles
fake pushes

extended entries
bad risk locations

It has:

no regime filter
no volatility filter beyond ATR existing
no session-awareness

So yes, it is functional.

But no, it is not smart enough yet to call “super efficient.”

This is still a basic breakout skeleton, not a strong autonomous trader.

Major problem 4: swing high / swing low logic is vulnerable to self-reference issues

This one matters.

The code computes:

rolling max of high
rolling min of low

Then the strategy checks whether the current close is above the current row’s swing_high or below swing_low.

Depending on how that rolling window is used, this can be logically weak because the current candle may already be part of the rolling calculation.

That can distort breakout detection.

Why this matters

A real breakout check usually compares current close to:

prior swing high excluding current candle
or
prior N-bar high shifted by 1

Same for lows.

If the rolling high includes the current candle’s own high, the breakout logic becomes less clean.

That is a serious detail.

This is exactly the kind of thing that makes a strategy look valid but behave weirdly.

So I would want the swing levels shifted:

previous swing high
previous swing low

Not same-row rolling values blindly used.

Major problem 5: order execution is not really complete trade management

This is very important.

The bot places a market order using ccxt. Fine.

But then:

stop loss is not placed on the exchange
take profit is not placed on the exchange
it only stores SL/TP internally
closing is based on polling current price and deciding internally

That means the bot is not using true attached protective orders.

Why that matters

If:

loop pauses
network lags
Replit sleeps
the code crashes
ticker fetch fails

Then your stored stop and take profit do nothing at the exchange level.

The exchange is not protecting the trade for you.

Now, for demo v1, this is acceptable as a prototype.

But it is still a weakness.

So the truth is:

This is not exchange-native position protection.
It is app-managed trade simulation/control.

Important difference.
